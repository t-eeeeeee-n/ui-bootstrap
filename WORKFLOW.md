# UI Bootstrap Workflow

ブランド系 DESIGN.md をベースに、自分のプロダクト用 UI モックを
Claude Code で高速に立ち上げるための手順書です。

このドキュメントが本ワークフローの **唯一のソースオブトゥルース** です。
同リポジトリの SKILL.md、`.claude/commands/*.md` はすべてここを参照します。

---

## 対象と前提

- **対象ユーザー**: プロダクトの UI モック（8画面前後）を Claude Code で素早く作りたい開発者
- **想定所要時間**: 30分〜90分（プロダクトの複雑さによる）
- **前提環境**: Node.js、Claude Code、Git
- **成果物**: ブランド風の一貫したデザインで統一された HTML モック群 + それを駆動する DESIGN.md / PRODUCT.md / SCREENS.md

---

## ワークフローの全体像

```
[Step 0] 事前準備
    ↓
[Step 1] DESIGN.md を入手
    ↓
[Step 2] DESIGN.md を脱ブランド化（自プロダクト化）
    ↓
[Step 3] PRODUCT.md を書く（プロダクト仕様・ダミーデータ正典）
    ↓
[Step 4] SCREENS.md を書く（画面仕様）
    ↓
[Step 5] パイロット1画面を先行生成 → 確認
    ↓
[Step 6] 残り画面を一気に生成
    ↓
[Step 7] 機械的チェック（残留ブランド語彙・ハードコード色）
```

各ステップに "完了条件" を設けています。次のステップに進む前に必ず確認してください。

---

## Step 0: 事前準備

### やること

1. プロダクトのざっくり仕様を固める
   - プロダクト名
   - 何をするサービスか（1〜2行）
   - 主要なユーザー
   - 作りたい画面の一覧（3〜10画面程度を推奨）
2. 作業用ディレクトリを作る

```bash
mkdir my-product-mock && cd my-product-mock
git init
```

### 完了条件

- [ ] プロダクト名が決まっている
- [ ] 画面一覧が書き出せている
- [ ] 空のリポジトリが用意できている

---

## Step 1: DESIGN.md を入手

### ブランドを決める

自プロダクトの性格に近いブランドを選びます。以下は代表例：

| プロダクトの性格 | 推奨ブランド |
|---|---|
| プロ向け、情報密度高め、ダーク | `linear.app`、`cursor`、`opencode` |
| AI プロダクト、温かみ | `claude` |
| ミニマル、メール系、モノスペース | `resend` |
| シネマティック、メディア系 | `elevenlabs`、`runway` |
| 明るく親しみやすい、B2C | `notion`、`airtable` |
| デザイン系、ビジュアル重視 | `figma`、`framer` |
| エンタープライズ、構造的 | `ibm` |

迷ったら `getdesign.md/<ブランド名>/design-md` をブラウザで開いて、
DESIGN.md タブから実物を眺めて決めます。

### コマンド

プロダクトディレクトリで：

```bash
npx getdesign@latest add <brand-name>
```

例：

```bash
npx getdesign@latest add linear.app
npx getdesign@latest add claude
npx getdesign@latest add cursor
```

※ **過去には `VoltAgent/awesome-design-md` の GitHub raw URL を直接 curl で取得する方法が流通していたが、エコシステムは `getdesign.md` に統合済みで、CLI経由が公式ルート。**

### もし CLI が使えない場合

ブラウザから以下を開き、DESIGN.md タブの中身をコピーしてローカルに保存：

```
https://getdesign.md/<brand-name>/design-md
```

### 完了条件

- [ ] プロジェクト直下に `DESIGN.md` が存在する
- [ ] ファイルサイズが 500 行以上ある（カラーパレットや Typography が実際に書かれている目安）

---

## Step 2: DESIGN.md を脱ブランド化

ダウンロード直後の DESIGN.md はブランド名と固有文脈を含んでいます。
これを **自プロダクト用に書き換える** 重要なステップです。

### 方針

- **保持する**: デザイントークン全般（色、フォント、サイズ、間隔、角丸、シャドウ）と視覚的な雰囲気
- **書き換える**: ブランド名、そのブランド固有のプロダクト文脈（用語、コンポーネント名）
- **追加する**: 自プロダクト固有のコンポーネント定義、プロダクト文脈

### プロンプト（Claude Code に投げる）

以下をそのまま渡し、`[...]` の部分だけ埋めます：

````markdown
このプロジェクトの DESIGN.md を以下の方針で書き換えて上書き保存してください。

## 保持するもの（変更禁止）
- すべてのデザイントークン（色のHEX値、フォント名、サイズ、間隔、角丸、シャドウ等）
- 視覚的な雰囲気
- コンポーネントのスタイル規約（Button、Input、Card、Table、Badge 等の見た目）

## 書き換えるもの
1. ブランド名 "[元のブランド名]" をすべて "[自プロダクト名]" に置換
2. そのブランド特有のプロダクト文脈を削除または自プロダクト用に読み替え
   （例: Linear なら issue / cycle / triage / roadmap 等）

## 追加するもの（自プロダクト固有のコンポーネント定義）
以下のコンポーネントのスタイル規約を DESIGN.md の作法に沿って追記してください：
[自プロダクト固有のコンポーネントを列挙。例:]
- [ コンポーネント名 ]: [ 用途と状態 ]
- [ コンポーネント名 ]: [ 用途と状態 ]

## プロダクトコンテキスト（追加セクション "Product Context" として冒頭付近に挿入）
- [自プロダクトの一行説明]
- [主要な価値提案]
- [対象ユーザー]
- [利用のカデンス（毎日 / 週次 / プロジェクト単位 等）]

作業後、変更の要約を 3〜5 行で報告してください。
````

### 完了条件

- [ ] Claude Code から「作業完了、要約は以下」の報告が来た
- [ ] `grep -niE "[元ブランド名の主要語彙]" DESIGN.md` の残留が許容範囲
- [ ] 冒頭付近に "Product Context" セクションが追加されている
- [ ] 新規コンポーネント定義が Section 内に追加されている

### よくある落とし穴

- **残留語彙を見逃す**: ブランド名を置換しただけで、そのブランド固有の用語（Linear なら issue、Stripe なら payment link など）が残ることがある。grep で必ず確認
- **トークンに手が入ってしまう**: 「自プロダクト風にトーンを変えた方が良いのでは」とエージェントが気を利かせて色を微調整することがある。完了後に色のHEX値が Step 1 の DESIGN.md と一致しているか一部抜き打ち確認する

---

## Step 3: PRODUCT.md を書く

DESIGN.md は **視覚判断** のための文脈、PRODUCT.md は **画面の中身** のための文脈として役割を分離します。

### 書くべき内容

1. **プロダクト概要** — 何をするサービスか、主要フロー
2. **ドメインモデル** — 主要なエンティティ、状態、ステートマシン
3. **用語集** — プロダクト固有の用語と意味
4. **ダミーデータ正典** — 全画面で使い回す統一ダミーデータ（ID、名前、日付等）
5. **キーボード操作・ショートカット** — プロ向け SaaS の場合は必須

### テンプレート

以下を `PRODUCT.md` として保存し、自プロダクトに合わせて埋めます：

````markdown
# [プロダクト名] — プロダクト仕様

（DESIGN.md の Section "Product Context" で視覚面の文脈は定義済み。
このファイルは画面の中身・用語・ダミーデータの正典として使う）

## プロダクト概要
[1〜3段落]

## 中核フロー
[主要なユーザージャーニーを 3〜7 ステップで]

## 主要なドメイン用語
| 用語 | 意味 |
|---|---|
| ... | ... |

## 状態遷移（該当する場合）
[フェーズや状態を持つ場合、表や箇条書きで]

## ダミーデータ正典（全画面で統一）

### メインとなる例示データ
- ID: `[...]`
- 名称: `[...]`
- 担当: `[...]`
- 作成日: `[...]`
[など]

### 一覧画面用の追加ダミー
- [列挙、10件前後を推奨]

### 対話・入力例（該当する場合）
- [具体的な文面をそのまま書く]

## キーボード操作前提（プロ向け SaaS の場合）
- `⌘K` [役割]
- `⌘Enter` [役割]
- [その他のショートカット]
````

### 完了条件

- [ ] PRODUCT.md が保存されている
- [ ] ダミーデータが具体的（「ユーザーA」ではなく「Kenta Yamamoto」のように）
- [ ] 画面生成時に "どの案件名を使うか" で迷わない粒度になっている

---

## Step 4: SCREENS.md を書く

各画面の構造を言語化します。図や Wireframe は不要、**箇条書きでレイアウトと内容を記述** すれば十分です。

### 書くべき内容

1. **共通レイアウト** — 全画面共通のヘッダー、サイドバー、上部バー等
2. **画面ごとの仕様** — レイアウト、主要コンポーネント、ダミーデータの指定

### テンプレート

以下を `SCREENS.md` として保存：

````markdown
# 画面仕様

## 共通レイアウト
[全画面共通の要素をここに記述]

ダミーデータは PRODUCT.md の「ダミーデータ正典」を唯一の出典として使う。

## 1. [ファイル名].html  `[URL パス]`
- [レイアウトの大枠]
- [主要セクション]
- [使うコンポーネント]
- [ダミーデータの指定]

## 2. [ファイル名].html  `[URL パス]`
[同上]

...

## index.html（ナビゲーション）
- ルートに `index.html` を配置
- 全画面への直接リンクを一覧表示
- 各リンクの右に画面の目的を一行で
````

### 書くときのコツ

- **一画面 8〜15 行** を目安に。長すぎると読まれない
- **レイアウトの構造は優先度順に**（上から下、左から右）
- **「このダミーデータを使う」と明示する** — Claude Code の裁量を減らす
- **最重要画面を特定し、[重要] のマーカーを付ける** — 後でパイロット生成の対象にする

### 完了条件

- [ ] SCREENS.md が保存されている
- [ ] 全画面分の記述がある
- [ ] パイロット生成候補の画面（最も複雑、最も多くのコンポーネントを使う）が特定できている

---

## Step 5: パイロット1画面を先行生成

最複雑画面を1枚だけ先行生成し、共有 CSS 資産（tokens.css / components.css）を同時に作らせます。ここが通れば残りの画面は量産できます。

### プロンプト

````markdown
DESIGN.md / PRODUCT.md / SCREENS.md を読み込み、
SCREENS.md の「[パイロット対象画面名]」に相当する HTML モックを生成してください。
これは残り画面の雛形となる「先行パイロット」です。

## 生成ファイル
- `mock/[画面ファイル名].html` — メイン画面
- `mock/tokens.css` — DESIGN.md の全デザイントークンを CSS 変数として集約（残り画面で共有）
- `mock/components.css` — 共通コンポーネントのスタイル（sidebar / topbar / button / input / card / badge / tabs / kbd / [自プロダクト固有のコンポーネント] 等）

tokens.css と components.css は以降の画面でも使い回す共有資産です。
[パイロット画面] 専用の書き方にせず、再利用可能に設計してください。

## 技術要件
- Tailwind CSS の CDN を使ってよいが、**色・サイズ・間隔・角丸・シャドウは必ず tokens.css の CSS 変数経由**で参照すること
- Tailwind の `bg-purple-600` 等のパレット直指定は禁止（DESIGN.md が唯一のソースオブトゥルース）
- アイコンは Lucide の SVG をインラインで
- 指定フォントは Google Fonts から読み込み
- デスクトップ幅（1440px）想定、レスポンシブ対応は不要

## 厳守事項
1. DESIGN.md のトークンのみ使用、発明禁止
2. PRODUCT.md のダミーデータ以外を使わない
3. 迷ったら [元ブランド] の作法を基準に

## 作業後に報告してほしいこと
1. DESIGN.md → tokens.css のマッピング表（主要なものだけ）
2. 追加した独自コンポーネントの実装方針
3. 迷った / 判断保留した箇所があれば列挙
````

### ブラウザで目視確認

`mock/[画面].html` を開き、以下を確認：

1. **全体の"らしさ"** — 初見でベースブランドに似た印象を受けるか
2. **色の階層** — 背景 / カード / ボーダー / テキストに明確な階層があるか
3. **アクセントカラーの使い方** — 発散せず、現在状態や CTA だけに絞れているか
4. **コンポーネントの差別化** — 類似要素（例: 複数バッジ種別）が適切に弁別できるか
5. **情報密度** — 対象ユーザー層に対して過不足ないか

### 完了条件

- [ ] ブラウザで開いて "DESIGN.md のブランドっぽい" と言えるレベル
- [ ] tokens.css と components.css が存在する
- [ ] `grep -rnE "var\(--" mock/ | wc -l` の結果が十分多い（50以上が目安）
- [ ] 違和感があればその点を言語化してフィードバック済み

### 違和感があった場合

「なんか違う」ではなく、**具体的に言語化して** 修正依頼を出します：

- ❌ 「Phase Bar が微妙」
- ✅ 「Phase Bar の現在フェーズが他のフェーズと同じに見えて区別できない。DESIGN.md の accent トークンで左ボーダーを引くか、背景色を一段明るくしてほしい」

## Step 6: 残り画面を一気に生成

パイロット画面が OK なら、残りを量産します。

### プロンプト

````markdown
mock/[パイロット画面].html が期待通り生成された前提で、SCREENS.md の残り画面を生成してください。

## 生成ファイル
- mock/[画面1].html
- mock/[画面2].html
- ...
- mock/index.html — 全画面へのナビゲーション

## 厳守事項
- **tokens.css と components.css を再利用**。パイロット画面と完全に共有
- 新規コンポーネントが必要になったら、勝手に増やさず components.css に追記する形で統合し、「何を足したか / DESIGN.md のどの規約から派生させたか」を報告すること
- PRODUCT.md のダミーデータ正典を全画面で使い回す
- 共通レイアウト（サイドバー、上部バー等）の構造と見た目は全画面で完全一致させる

## 画面ごとの特に気をつけたい点
[各画面で特に注意したい点を 1〜2 行で]

## 作業後に報告してほしいこと
1. 画面ごとに使った主要コンポーネントの一覧
2. tokens.css / components.css に追加があった場合、その内容と理由
3. SCREENS.md の記述と実装で乖離した点があれば列挙
````

### 完了条件

- [ ] 全画面の HTML ファイルが生成されている
- [ ] index.html から全画面に飛べる
- [ ] components.css の追加があった場合、内容が妥当

---

## Step 7: 機械的チェック

画面がすべて出揃ったら、プロジェクトルートで以下を実行：

```bash
# 1. 残留ブランド語彙チェック
grep -rniE "\b[元ブランド名]\b|\b[そのブランド固有の語彙1]\b|\b[語彙2]\b" mock/ DESIGN.md PRODUCT.md SCREENS.md

# 2. Tailwind パレット直指定の混入チェック（あってはならない）
grep -rnE "(bg|text|border|ring|from|to|via)-(red|blue|green|yellow|purple|violet|orange|pink|amber|slate|gray|zinc|neutral|stone)-[0-9]+" mock/

# 3. CSS 変数が実際に使われているかの気配チェック
grep -rnE "var\(--" mock/ | wc -l
```

### 判定基準

1. **残留語彙**: ヒット 0 件が理想。残っていた場合は該当行を修正
2. **パレット直指定**: ヒット 0 件が必須。あれば tokens.css 経由に置換
3. **CSS 変数利用**: 画面数 × 10 以上を目安に

### 完了条件

- [ ] 3つのチェックすべて判定基準を満たす
- [ ] または差分を把握しており、許容範囲と判断している

---

## 仕上げ

1. `README.md` を作り、「どう作ったか」を記録
2. 全画面のスクリーンショットを撮って確認
3. Git でコミット：

```bash
git add .
git commit -m "UI mock bootstrap: [ブランド名]ベース, [プロダクト名]用"
```

## トラブルシューティング

### DESIGN.md が薄すぎる（行数が少ない）

入手したブランドの DESIGN.md が手薄な場合、Claude Code に「このDESIGN.mdを読んで、不足しているセクション（Typography、Spacing、Component specs 等）を補強して」と追加作業を依頼する。元のトークンと矛盾しないことだけ指示する。

### 画面がのっぺりする

多くの場合、背景 / カード / 区切りの **surface 階層が使われていない**。DESIGN.md の surface 系トークン（surface-0, surface-1, surface-2 等）が tokens.css に定義されているか確認し、使われていなければ指示して修正させる。

### 画面ごとにトーンがばらつく

`components.css` が画面ごとに独自拡張されている疑い。`git diff` で components.css の差分を見て、パイロット画面生成後に追加された行が画面固有のハックになっていないか確認する。

### パイロット画面が的外れ

DESIGN.md の脱ブランド化（Step 2）が不十分な可能性が高い。DESIGN.md を読み直し、ブランド固有の語彙が残っていないか確認。残っていたら Step 2 からやり直す。

## このワークフローの設計思想

1. **3ファイル分離**: DESIGN.md（見た目）/ PRODUCT.md（中身）/ SCREENS.md（構造）の役割を明確に分ける。Claude Code が参照先を迷わない
2. **パイロット → 量産の2フェーズ**: 最複雑画面で共有資産を固めてから横展開。やり直しコストを最小化
3. **トークン経由の徹底**: Tailwind の直接パレットを禁じ、すべて CSS 変数経由で書かせる。DESIGN.md を書き換えるだけで全画面の印象を一気に変えられる状態を作る
4. **機械的チェック**: grep で残留とハードコードを検出。レビューを属人化させない
